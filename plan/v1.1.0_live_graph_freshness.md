# Feature Plan: v1.1.0 — Live Graph Freshness & Daemon Reindex

Lead feature of the 1.1.0 release. Covers GitHub issues
[#28](https://github.com/Warhorze/sql-code-graph/issues/28) (silent stale graph under
the single-writer lock), [#30](https://github.com/Warhorze/sql-code-graph/issues/30)
(freshness / staleness signal), and
[#29](https://github.com/Warhorze/sql-code-graph/issues/29) (MCP lifecycle + reindex
through the running server).

> Author: architect-planner · Date: 2026-05-31 · Status: drafted, ready for
> plan-reviewer → sprint-planner PR breakdown.

---

## Summary

While `sqlcg mcp start` holds the KuzuDB single-writer lock, any second process that
opens the DB (`index`/`reindex`/`db reset`) fails with "Database is locked". The git
hooks run `sqlcg reindex … || true`, so the failure is swallowed and the graph silently
goes stale — the worst failure mode for a correctness tool. This feature (a) makes
staleness *visible* (cheap, low-risk), then (b) lets the running daemon apply the
incremental delta *itself* so reindex never contends for the lock (involved, the real
fix).

**Process model (verified):** the server is a single process running one `anyio` event
loop ([`server.py`](src/sqlcg/server/server.py) `main` → `anyio.run`). All MCP tools call
the module-level singleton `_backend` synchronously inside the loop thread. There is no
existing IPC, no PID file, and no second thread. A `--notify` reindex must hand work to
*that* event loop, because it is the process holding the write lock.

**Constant alignment (verified):** the DB path default is owned by
[`KuzuConfig`](src/sqlcg/core/config.py) line 17 (`Path.home() / ".sqlcg" / "graph.db"`),
read via `get_db_path()` (line 65), overridable by `SQLCG_DB_PATH`. Any PID/socket file
**must** be derived from the resolved `get_db_path()`, never hardcoded to `~/.sqlcg`, so
that two servers on two DBs do not collide on one control file.

## Scope

### In Scope

- A shared **freshness helper** computing `indexed_sha`, `head_sha`,
  `stale_by_commits`, `dirty` from a repo root + the stored SHA.
- Surfacing freshness in `sqlcg db info` (CLI) and the MCP `db_info` tool.
- An optional compact `freshness` block on the primary lineage tool result(s).
- `sqlcg index --include-working-tree` to fold uncommitted changes.
- `sqlcg mcp status` / `stop` / `restart` lifecycle commands (JSON output).
- A **control channel** (Unix domain socket) so a live server can be discovered,
  queried, stopped, and asked to reindex itself.
- `sqlcg reindex … --notify`: when a server is live on the same DB, route the
  incremental delta *through the server* instead of opening the DB from a second
  process. Fall back cleanly to the current direct-write path when no server is live.
- Git hook update so `post-checkout`/`post-merge` use `--notify` and surface a one-line
  stderr cue when they cannot reach the server (kills the "silent" failure mode).

### Non-Goals (explicit boundaries)

- **No lock lease / reader-writer split inside KuzuDB.** Verified there is no API for the
  server to drop and re-take the write lock mid-session safely. The daemon-writer
  approach (server applies the delta) is the chosen fix and supersedes this idea from
  #28's "suggested fixes". Do not attempt a lock hand-off.
- **No Windows named-pipe / Windows IPC.** Unix domain socket only. Windows lifecycle
  (`status`/`stop`/`restart`) is **deferred to v1.2** (see Deferred). On Windows the
  hooks keep the current direct-write + `|| true` behaviour plus the staleness cue.
- **No multi-client connection counting fidelity.** `connected_clients` is reported
  best-effort (see Risk R4); stdio MCP has exactly one client by transport design.
- **No Neo4j freshness.** `set_indexed_sha`/`get_indexed_sha` already
  `raise NotImplementedError` on [`Neo4jBackend`](src/sqlcg/core/neo4j_backend.py)
  (lines 187–199). Freshness reporting guards against that and reports `indexed_sha:
  null` for Neo4j; daemon reindex is Kuzu-only.
- **No new daemonisation / detach.** The server is still launched by the editor as
  today (`sqlcg mcp start` over stdio). We add discovery + a control socket to the
  *existing* process; we do not add `--daemon`/fork.

## Design

### A. Freshness helper (shared, pure, testable)

New module [`src/sqlcg/core/freshness.py`](src/sqlcg/core/freshness.py):

```python
@dataclass(frozen=True)
class Freshness:
    indexed_sha: str | None      # from backend.get_indexed_sha()
    head_sha: str | None         # git rev-parse HEAD (None if not a git repo)
    stale_by_commits: int | None # git rev-list --count indexed_sha..head_sha
    dirty: bool                  # git status --porcelain non-empty
    branch: str | None           # git rev-parse --abbrev-ref HEAD (for the human cue)

def compute_freshness(root: Path, indexed_sha: str | None) -> Freshness: ...
def render_freshness_line(f: Freshness) -> str:
    # "indexed at <sha8> (N commits behind HEAD, working tree dirty)"
```

Rules:
- All git calls go through one private `_git(root, *args)` that returns `None` on any
  failure (matches the defensive pattern in `git_delta.py` / `watcher.py`). Never raise.
- `stale_by_commits` is computed with `git rev-list --count <indexed>..<head>`. If
  `indexed_sha` is unknown to git (rebased away, shallow clone), fall back to `None`
  (not 0) so consumers can say "unknown" rather than falsely "fresh".
- `dirty` = `git status --porcelain` produces any line (restricted to tracked `.sql`
  changes is a v1.2 refinement — v1.1 reports whole-tree dirty, documented).
- The helper does **not** import the backend; the caller passes `indexed_sha` in. This
  keeps it unit-testable with a fake git repo and no Kuzu.

### B. `DbInfoResult` extension (#30)

Add to [`DbInfoResult`](src/sqlcg/server/models.py): `indexed_sha`, `head_sha`,
`stale_by_commits`, `dirty` (all optional). The MCP `db_info` tool computes them via the
freshness helper using the **repo root recorded on the `Repo` node** (the indexed
`path`). When multiple repos are indexed, report freshness for each in a `repos: list`
sub-block; when one, inline it. CLI `db info` prints `render_freshness_line`.

### C. Optional freshness block on tool results (#30)

Add an optional `freshness: FreshnessBlock | None` to the result model of
`trace_column_lineage` (and `get_upstream_dependencies` / `get_downstream_dependencies`
if cheap). Populate it once per call from the cached server-side freshness (computed at
server start + refreshed on each notify-reindex — do **not** shell out to git on every
tool call; that adds latency to the hot path). Gate behind a server flag/env
(`SQLCG_FRESHNESS_ON_RESULTS`, default off) to avoid bloating every response.

### D. Control channel + lifecycle (#29)

**Transport: Unix domain socket**, path derived from the DB path:
`get_db_path().with_suffix(".sock")` (so `~/.sqlcg/graph.db` → `~/.sqlcg/graph.db.sock`).
A sibling `.pid` file holds the PID + db_path + start-time for `status` when the socket
is stale. Socket chosen over signal+file (race-prone, no reply channel) and over
named pipe (one-directional); a socket gives a request/response JSON protocol and works
inside the existing `anyio` loop (`anyio.create_unix_listener`).

**Server side** (in the existing process, started from `server.main`):
- On start: write `<db>.pid`, bind `<db>.sock`, spawn a control-socket task on the same
  `anyio` event loop alongside the stdio MCP loop.
- Protocol (newline-delimited JSON, one request → one response):
  - `{"op": "status"}` → `{running, pid, db_path, indexed_sha, head_sha,
    stale_by_commits, connected_clients, uptime}`.
  - `{"op": "stop"}` → graceful: finish in-flight tool call, close backend, unlink
    socket+pid, exit 0.
  - `{"op": "reindex", "root", "from", "to", "dialect"}` → run `resync_changed` **on the
    server's backend** (it already holds the write lock), return the summary dict, then
    refresh cached freshness. This is the lock-contention fix.
- On exit (including `finally` in `main`): unlink `.sock` and `.pid`.

**Reindex-through-server execution model (critical):** `resync_changed` is synchronous
and CPU/IO heavy. It must **not** run on the event-loop thread (would block all tool
calls). Run it via `anyio.to_thread.run_sync`, and serialise it behind an
`asyncio.Lock`/`anyio.Lock` so two notifies cannot mutate the single Kuzu connection
concurrently (the backend connection is not thread-safe — see ARCHITECTURE_REVIEW.md
finding 3.3 on shared-state concurrency). Tool calls during a reindex either queue or
return a "reindex in progress" freshness note — pick *queue* for correctness.

**CLI side** (new commands in [`mcp.py`](src/sqlcg/cli/commands/mcp.py)):
- `sqlcg mcp status` → connect to `<db>.sock`, send `status`, print JSON. If no socket
  but a `.pid` exists with a live PID, report `running:true, degraded:"socket
  unavailable"`. If neither, `{"running": false}`.
- `sqlcg mcp stop` → send `stop`; wait for socket to disappear; fall back to SIGTERM on
  the PID-file PID if the socket is unresponsive.
- `sqlcg mcp restart` → `stop` then re-exec is **out of our control** (the editor owns
  the stdio process lifecycle). Restart instead means: `stop` the current server and
  print guidance, OR — if launched standalone — `stop` + spawn. Decision: v1.1 `restart`
  = `stop` + clear instruction that the client must respawn. Document this honestly;
  do not pretend to restart a process we did not parent. (Flagged as a scope nuance, not
  a blocker.)

**`reindex --notify` flow:**
1. Resolve `<db>.sock` from `get_db_path()`.
2. If socket connects → send `reindex` op, print returned summary, exit 0. Graph updated
   with **zero lock contention**.
3. If no live socket → fall through to the **existing direct-write path** unchanged.
4. If a socket exists but the server errors → surface the error on stderr (non-silent),
   exit non-zero (the hook's `|| true` keeps the checkout non-fatal but the cue is now
   visible).

### E. Git hook update (#28 visibility)

Update `_HOOKS` in [`git.py`](src/sqlcg/cli/commands/git.py) to pass `--notify` and to
echo a one-line stderr cue when reindex cannot reach a server *and* the direct write is
locked:
```
sqlcg reindex --from "$1" --to "$2" "$(git rev-parse --show-toplevel)" \
  --dialect auto --quiet --notify \
  || echo "sqlcg: graph not updated (server busy/locked) — run 'sqlcg mcp status'" >&2
```
The `|| true` semantics are preserved (checkout never blocks) but the failure is no
longer silent. This single line is the cheapest, highest-value mitigation of #28 and
ships in Phase 1 even before `--notify` exists (the flag is added in a later phase; the
hook text lands with a TODO-free no-op `--notify` that is accepted-and-ignored until the
socket exists — **do not** ship an accepted-but-silent flag; sequence so the hook change
lands in the same PR as `--notify` parsing).

- Acceptance: `db info` on an indexed-then-advanced repo prints
  `indexed at <sha8> (2 commits behind HEAD, working tree dirty)`. On an empty DB prints

**Step 1.3** — Extend `DbInfoResult` + MCP `db_info` tool.

Phase 1 ships independently and is a complete, mergeable answer to #30.

### Phase 2 — Working-tree indexing (LOW–MEDIUM, optional, independent of IPC)

**Step 2.1** — `sqlcg index --include-working-tree`.
- Files: [`index.py`](src/sqlcg/cli/commands/index.py); indexer already walks the tree,
  so this is mainly: do not restrict to committed state, and record a sentinel
  `indexed_sha = "<head>+dirty"` (or store `dirty=true` alongside) so freshness can show
  "indexed with working-tree changes".
- Acceptance: with an uncommitted edit, `index --include-working-tree` then `db info`
  shows the working-tree marker; without the flag the edit is not reflected.
- **Defer to v1.2 if it balloons**: per-file dirty tracking / partial working-tree
  deltas. v1.1 scope is the whole-tree flag only.

### Phase 3 — Control socket + lifecycle (MEDIUM–HIGH RISK / HIGH REWARD — real #29/#28 fix)

**Step 3.1** — Control-file paths derived from `get_db_path()`; `.pid` writer/reader.
- Files: new `src/sqlcg/server/control.py` (paths + pid file helpers).
- Acceptance: paths derive from `SQLCG_DB_PATH` override, never hardcoded `~/.sqlcg`.

**Step 3.2** — Server-side control-socket task on the existing `anyio` loop; `status`
and `stop` ops; socket+pid cleanup in `server.main` `finally`.
- Files: [`server.py`](src/sqlcg/server/server.py), `server/control.py`,
  [`tools.py`](src/sqlcg/server/tools.py) (expose cached freshness + connection count).
- Acceptance: e2e starts a server, `mcp status` returns running+pid+db_path; `mcp stop`
  causes the socket+pid to disappear and the process to exit 0.

**Step 3.3** — `sqlcg mcp status` / `stop` / `restart` CLI.
- Files: [`mcp.py`](src/sqlcg/cli/commands/mcp.py).
- Acceptance: status JSON matches #29 fields; stale-socket-but-live-pid reports
  `degraded`; no server reports `{"running": false}`; `restart` documents the
  client-respawn caveat.

### Phase 4 — Reindex through the running server (HIGH RISK / HIGH REWARD — closes #28)

**Step 4.1** — `reindex` op on the server: run `resync_changed` via
`anyio.to_thread.run_sync` under a serialising lock; refresh cached freshness; return
summary. **No `# TODO` in the success branch** — the op must call the real
`Indexer.resync_changed` (verified signature at
[`indexer.py`](src/sqlcg/indexer/indexer.py) line 428).
- Files: `server/control.py`, [`server.py`](src/sqlcg/server/server.py).
- Acceptance: integration test — server live + locked; send `reindex` op; assert the
  graph reflects the new commit **and** the server kept serving (a tool call before and
  after succeeds, no "Database is locked").

**Step 4.2** — `sqlcg reindex … --notify` flag + fallback.
- Files: [`reindex.py`](src/sqlcg/cli/commands/reindex.py).
- Acceptance: with a live server, `reindex --notify` updates the graph and never opens a
  second DB handle (assert no lock error; assert via the server summary, not the local
  path). With no server, `--notify` is a no-op that falls through to the existing
  direct-write path (existing reindex tests still pass).

**Step 4.3** — Git hook uses `--notify` + stderr cue (lands with 4.2, same PR).
- Files: [`git.py`](src/sqlcg/cli/commands/git.py).
- Acceptance: e2e — server live, simulate `post-checkout`: hook updates the graph via
  the socket, exit 0, no lock error. Server down + DB lockable: direct write succeeds.
  Server up but reindex errors: stderr cue printed, checkout still non-fatal.

## Risks and Mitigations

| ID | Risk | Sev | Mitigation |
|----|------|-----|------------|
| R1 | `resync_changed` on the event-loop thread blocks all tool calls during a reindex | HIGH | Run via `anyio.to_thread.run_sync`; serialise notifies behind one lock; tool calls queue. |
| R2 | Two concurrent notifies mutate the single Kuzu connection (not thread-safe; ARCH 3.3) | HIGH | Single serialising lock around the reindex op; the connection is touched by exactly one thread at a time. |
| R3 | Stale `.sock`/`.pid` after a crash makes `--notify`/`status` think a dead server is alive | MED | `status`/`notify` connect-and-probe the socket; on connection refused, treat as not-running, clean stale files, fall through to direct write. |
| R4 | `connected_clients` is not truly knowable over stdio MCP | LOW | Report best-effort (stdio = 1 by transport); document as approximate. Not an acceptance gate. |
| R5 | `restart` cannot re-parent an editor-spawned stdio process | MED | v1.1 `restart` = `stop` + explicit client-respawn guidance; documented, not faked. |
| R6 | Socket security — any local user could send `stop`/`reindex` | MED | Socket file mode 0o600, owner-only, under the user's `~/.sqlcg`. Document; no auth token in v1.1. |
| R7 | Windows has no Unix domain sockets | MED | Out of scope (deferred to v1.2); Windows keeps direct-write + staleness cue. Guard socket code behind a platform check so import never breaks on Windows. |

## Deferred to v1.2 (flagged to prevent scope balloon)

- **Windows IPC** (named pipes) for lifecycle + notify-reindex.
- **Per-file / SQL-only dirty tracking** and partial working-tree deltas (v1.1 reports
  whole-tree dirty only).
- **Socket auth token** for multi-user machines (R6).
- **True `restart`** that re-parents the stdio process (R5) — needs a launcher model the
  editor does not currently give us.
- **Freshness on every tool result by default** — v1.1 ships it gated/off to protect the
  hot-path latency; default-on is a v1.2 decision after measuring overhead.
- **Reader/writer lock lease inside KuzuDB** — explicitly rejected (no safe API); the
  daemon-writer path supersedes it.

### Blocking Questions

None blocking. Two confirmations the developer should not guess on (resolve at PR time,
not now):

1. **`stale_by_commits` direction semantics** — confirmed as commits HEAD is *ahead of*
   the indexed SHA (`indexed..HEAD`), i.e. "graph is N commits behind". Documented above.
2. **`restart` honesty** — v1.1 cannot re-parent the editor's stdio process; `restart`
   degrades to `stop` + guidance (R5). If the user wants a true restart, that is a v1.2
   launcher item. Proceeding with the documented degradation unless the user objects.
