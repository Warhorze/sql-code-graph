carries its real source `file:line` + producing `expression` (#31), confidence is `1.0`

## Scope

### In Scope

- **#31**: persist source location and expose `file`, `line`, `expression` on every lineage node in BOTH the MCP `trace_column_lineage` output and the CLI lineage path.
- **#32**: replace the blanket `0.7` with a meaningful model — `1.0` (or omitted) for deterministic edges, `<1.0` + `reason` only for inferred edges; document the scale in the skill / mcp-best-practices boundary.
- **#33 part 1**: bring `analyze impact` (and `unused`) to parity with `upstream`/`downstream` — `NoiseFilter.from_config()` + de-dup + `--raw` opt-out.
- **#33 part 2**: tag every `SqlTable` node with `kind ∈ {table, cte, derived, external}`; exclude `cte`/`derived` from `upstream`/`impact` by default (with a flag to include).

### Non-Goals (deferred to v1.2 — see "Deferred" section)

- **Per-edge sub-statement line numbers** (the exact line of a single projection inside a wide SELECT). sqlglot does not populate AST-node positions; recovering them means re-tokenizing per node — a hot-path cost. Statement-start line is the v1.1.0 granularity.
- **Clickable `file:line` rendering in the MCP transport** beyond emitting the `file` + `line` fields as data. Terminal hyperlink escapes in the CLI are in scope; MCP just returns the fields.
- **`external` kind inference from a catalog/allowlist.** v1.1.0 derives `kind` structurally (cte/derived vs table); `external` is reserved in the enum but only assigned where structurally unambiguous (e.g. a `<output>` synthetic sink is not `external`). Full external/source classification is v1.2.
- **Confidence calibration** (measured false-positive rates). The scores stay explicitly UNCALIBRATED, consistent with `Judgement.confidence` docs in [`models.py:22-28`](../src/sqlcg/server/models.py).

## Design

### Line granularity decision (resolves the #31 investigation)

sqlglot expression nodes have empty `.meta`; only tokens carry positions. The cheap,
correct unit is the **statement start line**, computed once per file from the tokenizer
stream by grouping tokens on `SEMICOLON` and taking the first token's `.line` per
statement. This is O(tokens) per file, done **once**, and maps cleanly onto the existing
`statement_index`. It does NOT touch `_extract_column_lineage` (the O(N²)-guarded path).

- `line` on a lineage node = the start line of the producing statement.
- `expression` = the statement SQL already stored on `SqlQuery.sql` (currently truncated to 500 chars; keep that truncation).
- `file` = `SqlQuery.file_path`.

All three are reachable by joining a `COLUMN_LINEAGE` edge to its `SqlQuery` via the
already-stored `query_id`. The only **new persisted datum** is the per-statement start
line.

### Schema changes (exact, with migration)

Re-index is the migration path (CLAUDE.md). The version gate in
[`reindex.py:167`](../src/sqlcg/cli/commands/reindex.py),
[`index.py:186`](../src/sqlcg/cli/commands/index.py), and
[`watch.py:36`](../src/sqlcg/cli/commands/watch.py) compares stored vs `SCHEMA_VERSION`
and **hard-fails with `Exit(1)`** telling the user to `db reset && db init && index`.
So a property addition is safe iff `SCHEMA_VERSION` bumps to invalidate old graphs.

**`SCHEMA_VERSION` MUST bump `"4" → "5"`** ([`schema.py:6`](../src/sqlcg/core/schema.py)).

Two changes to [`schema.cypher`](../src/sqlcg/core/schema.cypher):

1. **#31 — add `start_line` to `SqlQuery`** (NOT to the edge — line is a property of the statement, and the edge already references the query via `query_id`):
   ```
   CREATE NODE TABLE SqlQuery (
       ... existing fields ...
       parsing_mode STRING,
       start_line INT64        -- NEW: 1-based start line of the statement in the file
   );
   ```
   Rationale for putting it on `SqlQuery` not `COLUMN_LINEAGE`: avoids per-edge bloat (one query has many edges), and the trace query already needs a query→file join to populate `expression`/`file`, so the line rides along on that same join. Keeps the indexer edge-row builder (hot path) untouched.

2. **#33 part 2 — `SqlTable.kind` already exists**; no DDL change needed for the column itself. The change is in **what value is written**: the indexer currently hardcodes `"TABLE"`. New allowed values: `table | cte | derived | external` (lowercase, new enum). DDL-defined views keep their existing `VIEW`/CREATE-kind value — confirm the enum reconciles DDL kinds (`TABLE`/`VIEW`) with the new lineage kinds; recommend `kind` stays the structural-role tag (`table`/`cte`/`derived`/`external`) and a separate existing path keeps DDL `TABLE`/`VIEW`. **Open design point for sprint-planner to lock** — see Blocking Questions.

No change needed to `COLUMN_LINEAGE`, `SqlColumn`, or any other table. #32 reuses the
existing `COLUMN_LINEAGE.confidence FLOAT` + `transform STRING` columns — no schema change,
only different values written + a new optional `reason` carried in the MCP model.

### Trace query change (#31)

[`queries.cypher`](../src/sqlcg/core/queries.cypher) `TRACE_COLUMN_LINEAGE` currently:
```
MATCH (dst:SqlColumn {id: $id})<-[r:COLUMN_LINEAGE]-(src:SqlColumn)
RETURN src.id, src.col_name, src.table_qualified, r.transform, r.confidence
```
Extend to join the producing query via `r.query_id`:
```
MATCH (dst:SqlColumn {id: $id})<-[r:COLUMN_LINEAGE]-(src:SqlColumn)
OPTIONAL MATCH (q:SqlQuery {id: r.query_id})
RETURN src.id AS id, src.col_name AS col_name, src.table_qualified AS table_qualified,
       r.transform AS transform, r.confidence AS confidence,
       q.file_path AS file, q.start_line AS line, q.sql AS expression
```
`OPTIONAL MATCH` keeps STAR_EXPANSION edges (whose `query_id` points at the originating
query) working and degrades to nulls if a query node is missing.

### Model changes (#31, #32)

[`models.py`](../src/sqlcg/server/models.py) `LineageNode` gains:
```python
line: int | None = Field(None, description="1-based start line of the producing statement")
expression: str | None = Field(None, description="SQL text of the producing statement (truncated)")
reason: str | None = Field(None, description="Set only when confidence < 1.0; why the edge is inferred")
```
`file` already exists; just stop hardcoding `None`. `confidence` stays `float | None`.

### Confidence model (#32)

| Edge origin | transform | confidence | reason |
|---|---|---|---|
| Plainly parsed `SELECT col [AS x]` resolved to a real source column | `SELECT` | **1.0** | None |
| `SELECT *` expanded via STAR_SOURCE | `STAR_EXPANSION` | **0.8** | `"star-expansion: columns inferred from source table schema"` |
| Schema-miss reduced edge | `UNKNOWN` | **0.5** | `"column not found in resolved schema"` |
| Scripting-mode fallback (Command body / regex DML) | (existing) | **0.3** | `"scripting-block fallback: column lineage approximate"` |
| Lineage extraction failed | `UNKNOWN` | **0.0** | `"lineage extraction failed at index time"` |

The single load-bearing change for #32 is **`base.py:500` `confidence=0.7` → `confidence=1.0`**
in `_lineage_node_to_edges._walk`. This is a one-line value change inside the parser but is
**not** in the perf-guarded section (it is the tree-walk emit, not `_extract_column_lineage`'s
column loop / scope build). It does not add per-edge work. The `0.8/0.5/0.3/0.0` values already
exist and are correct — only the dominant fact path is wrong today. `reason` is attached in the
MCP layer (tools.py), keyed off `transform`/`confidence`, so no parser data-model churn.

Documentation: update `_BOUNDARY` in [`skill.py:39-47`](../src/sqlcg/server/skill.py). Today it
claims "every other tool returns facts" while trace returns 0.70 on everything — directly
self-contradictory. New text: deterministic lineage edges are facts (`confidence=1.0`/omitted);
`confidence<1.0` flags an inferred edge and always carries a `reason`.

### CLI parity (#33 part 1)

In [`analyze.py`](../src/sqlcg/cli/commands/analyze.py): apply the **existing**
`NoiseFilter.from_config()` + a de-dup pass + `--raw` opt-out to `impact` (and `unused`),
matching the pattern already in `upstream`/`downstream`. `impact` returns `q.id, q.kind`
(query rows, not column ids) so it needs a query→table noise check via the query's
`target_table`/file, not `_col_id_to_table`. De-dup `upstream`/`downstream` results
(currently a flat `LIMIT 100` list that can repeat a node reached by multiple paths) by
`id`.

### Node-kind exclusion (#33 part 2)

- Indexer: when building table rows, tag CTE-derived destinations (`TableRef` minted from
  `cte_alias` at base.py:960) as `kind="cte"`, named-subquery/derived tables as
  `kind="derived"`, real tables as `kind="table"`. The `<output>` synthetic sink
  (base.py:494) should be filtered out entirely rather than tagged. This requires the
  parser to mark which `TableRef`s are CTE/derived (carry a `role` through `StarSource`/
  edge metadata or a side-set) — **the indexer must learn the role without the parser
  hot path re-walking the AST.** Recommended: emit the role from where the synthetic
  `TableRef` is *already* constructed (base.py:960 knows it is a CTE; the `<output>` site
  knows it is a sink), at no extra traversal cost.
- Query layer: `upstream`/`impact` default Cypher gains `WHERE src.kind IN ['table','external']`
  (or `NOT IN ['cte','derived']`); a `--include-intermediate` flag drops the filter.
- MCP `trace_column_lineage`: surface the source table's `kind` on `LineageNode.kind` is
  already "column"; add the *table* kind via the table_qualified node so CTE-origin edges
  are labelled, not silently dropped (trace shows the full chain; analyze excludes).

### Blocking Questions

1. **`SqlTable.kind` — repurpose vs add a column (VERIFIED: repurpose is safe).**
   The `STALE_VIEWS` query ([`queries.cypher`](../src/sqlcg/core/queries.cypher)) matches
   `(v:SqlTable {kind: 'VIEW'})`, but the indexer writes **`kind: "TABLE"` for every defined
   table including DDL views** ([`indexer.py:933`](../src/sqlcg/indexer/indexer.py)) — confirmed
   by reading the defined-table row builder. So `kind == 'VIEW'` is never written today and
   `STALE_VIEWS` is effectively dead (a latent bug, separate from this cluster). Consequence:
   repurposing `kind` to the structural-role enum `{table, cte, derived, external}` breaks
   **nothing currently working** — no DDL `+column` is needed for #33, and Step 3.1 stays a
   value change only.
   - **Recommendation for sprint-planner**: repurpose `kind` to the role enum. Note the dead
     `STALE_VIEWS` query in the sprint plan; either fix it to use a real VIEW signal (the
     `SqlQuery.kind == 'CREATE_VIEW'` path at indexer.py:944 is the authoritative VIEW marker)
     or drop it. **Confirm no other consumer depends on `SqlTable.kind` values before locking.**
   - This is the only open point that could change #33's schema surface, and the verified
     answer is "no extra column." Not a stop for #31/#32.
