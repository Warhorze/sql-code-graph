# Feature Plan: Dialect-Default & Query-Config Fix Batch

> Source: live DWH verification 2026-06-10 (post graph-health sprint, master at
> v1.13.0, 1,379 files indexed). All numbers below were **measured on the real
> DWH corpus**. See [`graph_health_sprint_postmortem.md`](../reports/graph_health_sprint_postmortem.md)
> — its "final measured state" table was measured on the ANSI **misparse** that
> Fix 1 corrects, so it understates the graph.

## Summary

Four independent fixes that restore correct lineage extraction and query-time
config resolution on the DWH. The headline bug: `sqlcg index <path>` with no
`--dialect` flag silently uses the **ANSI** parser instead of the repo's
declared dialect, because `--dialect` defaults to `None` and only the literal
string `"auto"` triggers `.sqlcg.toml` resolution. On the DWH this dropped 77%
of ETL edges onto bare-name nodes and produced an empty `analyze upstream`.

## Measured impact of the headline bug (Fix 1)

Indexing the DWH **with** `--dialect snowflake` vs. the silent ANSI default:

| Metric | ANSI default (today) | `--dialect snowflake` |
|--------|----------------------|-----------------------|
| ETL edges on bare-name nodes | 19,561 / 25,356 (77%) | — |
| `analyze upstream ba.wtfa_kassatransactie.ma_bedrag_inc` | **no results** | results |
| Strict edge health | 74% | 83% |
| Scoped edge health | 84% | 97% |
| Contradicted edges | 688 | 242 |
| Failed files | 146 | 1 |
| Files with column lineage | 824 | 902 |

# Fix 1 — `--dialect` defaults to `"auto"` (critical, tiny)

## Problem

[`index.py:30-32`](../../src/sqlcg/cli/commands/index.py) declares
`dialect: str | None = typer.Option(None, ...)`. Resolution only happens for the
literal sentinel:

```python
if dialect == "auto":          # index.py:109
    dialect = get_dialect(path)
```

So the documented runbook `sqlcg index <path>` (no flag) passes `None`
straight through. `None` → the parser pool selects ANSI. The DWH's
`.sqlcg.toml` says `dialect = "snowflake"` but is never read. Same pattern in
[`reindex.py:36-38, 84-85`](../../src/sqlcg/cli/commands/reindex.py).
[`watch.py:21-23`](../../src/sqlcg/cli/commands/watch.py) is worse — it has **no
`"auto"` resolution at all** and passes `dialect` straight to `index_repo`
(line 47) and into `WatchJobManager` / `SqlFileEventHandler` (lines 50-53).

## Design

Change the default value of the `--dialect` option from `None` to `"auto"` in
all three commands, and ensure each command resolves `"auto"` →
`get_dialect(path)` **before** the dialect is consumed.

### Where resolution must land in each file

- **`index.py`** — the `if dialect == "auto": dialect = get_dialect(path)` line
  already exists at line 109 and is correctly placed **before**
  `_try_route_index_via_server` (the "Bug A" comment, lines 105-108: the server
  WriterRequest must carry a concrete dialect, never `"auto"`). Only the option
  default (line 31) changes. Verify the resolved value reaches `_run_index` →
  `indexer.index_repo`.
- **`reindex.py`** — the resolution at line 84 is already **before**
  `_try_route_reindex_via_server`. Only the option default (line 37) changes.
  Note `reindex` has its own internal full-index fallback (lines 157-163) — the
  resolved dialect flows there too, confirm.
- **`watch.py`** — add resolution. Import `get_dialect` (and `config_file_present`
  for the banner if matching index's UX) and resolve **before** the first
  `indexer.index_repo(path, dialect, backend)` call **and** before the resolved
  `dialect` is passed to `WatchJobManager(indexer, backend, dialect)` and
  `SqlFileEventHandler(..., dialect=dialect)`. There is no server-routing path in
  watch, so placement is simply "right after `path` is known and before the first
  consumer".

### Behavioural change for repos with **no** `.sqlcg.toml`

`get_dialect(path)` already falls back to `"snowflake"` when no `.sqlcg.toml`
exists. So after the fix, a config-less repo gets the **snowflake** parser
instead of ANSI. **This is the correct and intended change**: the existing CLI
banner already tells the user
`"using defaults (snowflake dialect, no aliases/prefixes)"`
([`index.py:150-155`](../../src/sqlcg/cli/commands/index.py)). The current
`None`→ANSI default **contradicts that banner**. Document this in the PR body and
in the postmortem. No config-less DWH exists in the test corpus to regress.

### Constraints (do not violate)

- The sentinel `"auto"` must **never** reach a `WriterRequest` / the server
  payload. Resolution stays before `_try_route_*_via_server` in `index`/`reindex`.
  (The existing "Bug A" comments document exactly this; keep them.)
- Do not pass schema sources into the parser. This is arg resolution only.

## Implementation Steps

**Step 1.1** — `index.py`: change option default `None` → `"auto"` (line 31).
- Files affected: [`index.py`](../../src/sqlcg/cli/commands/index.py)
- Acceptance: running `sqlcg index <repo-with-snowflake-toml>` (no `--dialect`)
  resolves to `"snowflake"` and the snowflake parser runs.

**Step 1.2** — `reindex.py`: change option default `None` → `"auto"` (line 37).
- Files affected: [`reindex.py`](../../src/sqlcg/cli/commands/reindex.py)
- Acceptance: same resolution for both explicit-SHA and standalone modes,
  including the no-stored-SHA full-index fallback.

**Step 1.3** — `watch.py`: change option default `None` → `"auto"` and add the
`if dialect == "auto": dialect = get_dialect(path)` resolution before the
initial index and before `WatchJobManager` / `SqlFileEventHandler` receive it.
- Files affected: [`watch.py`](../../src/sqlcg/cli/commands/watch.py)
- Acceptance: the initial index and every incremental re-index in watch mode use
  the resolved dialect.

**Step 1.4** — Update help text where it says only `SQL dialect` (watch.py:22) to
match index/reindex (`"(or 'auto' to read from .sqlcg.toml)"`).

## Test Strategy

- **Regression test that would have caught this (required):** an integration
  test that indexes a tiny fixture repo whose `.sqlcg.toml` declares
  `dialect = "snowflake"` and which contains a file using `USE SCHEMA <s>;`
  followed by an `INSERT INTO t SELECT ... FROM <bare>` (snowflake-only USE
  SCHEMA qualification). Invoke the `index` command **with no `--dialect` flag**,
  then assert the resulting graph has **schema-qualified** SqlTable / column-id
  node IDs (e.g. `s.t.col`), not bare ones. With the old `None` default this test
  fails (ANSI parse → bare nodes); with the fix it passes. Assert on observable
  node IDs in the DB, not "no exception".
- **Resolution-reaches-parser unit test:** index/reindex/watch each resolve
  `"auto"` to the config dialect and pass that concrete string to
  `index_repo` / the parser pool. Patch/spy the indexer entry point and assert
  the dialect argument equals `"snowflake"` (or the fixture's declared dialect),
  proving the resolved value — not the sentinel — reaches the parser.
- **`watch_cmd` resolution unit test (new code, no existing coverage):**
  `watch.py` currently has **zero** `"auto"`-resolution logic — Step 1.3 adds it
  from scratch, unlike `index`/`reindex` where only the option default changes.
  Mock `WatchJobManager`, `SqlFileEventHandler`, and `Observer` (so the test does
  not block on the watch loop) and the indexer's `index_repo`; invoke
  `watch_cmd(path, dialect="auto")` against a `.sqlcg.toml` declaring
  `dialect = "snowflake"`; assert (a) `index_repo` is called with
  `dialect="snowflake"`, not `"auto"`, and (b) both `WatchJobManager(...)` and
  `SqlFileEventHandler(..., dialect=...)` receive `"snowflake"`. This is the
  primary regression guard for watch.py since it has no graph-output test.
- **Config-less repo test:** a repo with no `.sqlcg.toml` resolves to
  `"snowflake"` (documents the intended behaviour change away from ANSI).
- **Sentinel-never-routed guard:** assert that when a server socket is present,
  the routed payload's `dialect` is the concrete resolved value, never `"auto"`
  (protects the existing Bug A fix).

## Acceptance Criteria
- [ ] `sqlcg index <repo>` with no `--dialect` reads `dialect` from the repo's
      `.sqlcg.toml`; a snowflake repo parses with the snowflake parser.
- [ ] `reindex` and `watch` exhibit the same default resolution.
- [ ] A repo with no `.sqlcg.toml` resolves to `snowflake` (matches the banner).
- [ ] The `USE SCHEMA` fixture produces schema-qualified node IDs in the graph.
- [ ] The routed server payload never contains `"auto"`.

# Fix 2 — `catalog load` applies schema aliases at load time

## Problem

`sqlcg catalog load columns.csv` (PR #71, v1.11.0,
[`catalog.py`](../../src/sqlcg/cli/commands/catalog.py)) loads
`INFORMATION_SCHEMA.COLUMNS` rows **verbatim**. The DWH CSV contains staging
schemas, so load creates **2,907 phantom table nodes** (1,847 `ba_tmp.*` +
1,060 `da_tmp.*` ≈ 41% of all 7,038 table nodes) with zero lineage edges, plus
~42k `HAS_COLUMN` rows on `ba_tmp` alone. They pollute `find table`, inflate the
"tables catalogued" metric, and waste ~40% of the 82s load time.

The alias map (`get_schema_aliases`) already maps `ba_tmp → ba`, `da_tmp → da`.
Folding schema names through it at load time makes `ba_tmp.x` rows upsert as
`ba.x`, the same canonical identity the lineage edges already use.

## Design decisions (resolve these explicitly — do not guess)

### D2.1 — Where does the alias map come from?

`catalog load` currently takes only a CSV path and opens the backend with
`get_backend()` ([`catalog.py:233`](../../src/sqlcg/cli/commands/catalog.py));
it has **no repo root**, so `get_schema_aliases(path)` has nothing to read.

**Decision (PR 2 ships before PR 3 per the table — implement standalone):**
add a small local helper in `catalog.py`, e.g. `_resolve_repo_root(backend) ->
Path`, that runs `SELECT path FROM "Repo" LIMIT 1` against the **already-open
backend handle** (`catalog_load` already does `with get_backend() as backend:`
at [`catalog.py:233`](../../src/sqlcg/cli/commands/catalog.py) — same pattern as
`_indexed_root(db)` in [`tools.py:276`](../../src/sqlcg/server/tools.py), just a
local copy since `tools.py` is server-side and `catalog.py` is CLI-side). Falls
back to `Path.cwd()` if no `Repo` row. Call `get_schema_aliases(root)` and pass
the result through to `apply_catalog_to_backend`.

**This does NOT reintroduce the cwd disease Fix 3 cures**: `catalog.py` reads
`Repo.path` from the graph via its own backend handle — the cwd fallback only
applies to un-indexed graphs, identical to `_indexed_root`'s contract. PR 2 and
PR 3 are therefore independent and can merge in either order without conflict.

**Follow-up for PR 3 (Fix 3's Step 3.4):** once PR 3
lands `resolved_repo_root()` / confirms the `_indexed_root(db)` pattern as the
canonical helper, **PR 3 should replace `catalog.py`'s local
`_resolve_repo_root(backend)` with a call to the shared helper** (or extract
`_indexed_root` to a shared location both `tools.py` and `catalog.py` import) —
do not leave two copies of `SELECT path FROM "Repo" LIMIT 1` in the codebase
after PR 3 merges. Grep for `_resolve_repo_root` as part of PR 3's pre-flight
grep-confirm step.

### D2.2 — Precedence must not be violated

The backend already enforces **ddl > information_schema > usage** for
`HAS_COLUMN` (`upsert_edges_bulk`) and `insert_table_nodes_if_absent` only fills
empty `kind`/`defined_in_file`. Folding `ba_tmp.x → ba.x` must **not** let an
information_schema-sourced `ba.x` overwrite a **ddl-sourced** `ba.x`. Because the
existing upsert already protects ddl rows, folding is safe **as long as folding
happens before the upsert** (i.e. the folded row enters the same
precedence-aware path). Add a test that proves a ddl-sourced `ba.x` column
survives a catalog load whose CSV contains `ba_tmp.x`.

### D2.3 — Dedup when both `ba.x` and `ba_tmp.x` exist in the CSV

After folding, both rows canonicalize to `ba.x`. The existing `seen_tables` /
`seen_col_ids` dedup sets in `load_catalog_csv`
([`catalog.py:89-91`](../../src/sqlcg/cli/commands/catalog.py)) already collapse
duplicates **keyed on the qualified name** — so folding *before* the dedup check
makes the dedup free: the first folded occurrence wins, the second is skipped.
Apply folding to `schema` immediately after the case-fold at
[`catalog.py:107`](../../src/sqlcg/cli/commands/catalog.py), before
`table_qualified` is built. Document that column metadata is assumed identical
between staging and canonical (true for `_tmp` mirrors); if it differs, first
wins — acceptable and noted.

## Implementation Steps

**Step 2.1** — Thread an alias map into `load_catalog_csv` /
`apply_catalog_to_backend`.
- Add a `schema_aliases: dict[str, str]` parameter (default empty dict for the
  zero-config / test path).
- Files affected: [`catalog.py`](../../src/sqlcg/cli/commands/catalog.py)

**Step 2.2** — In the row loop, fold the schema before building
`table_qualified`: `schema = schema_aliases.get(schema, schema)`. This is the
single behavioural line; dedup and precedence flow unchanged.

**Step 2.3** — In `catalog_load` (the CLI command), implement
`_resolve_repo_root(backend)` (D2.1: `SELECT path FROM "Repo" LIMIT 1` against
the open backend handle, else `Path.cwd()`), call `get_schema_aliases(root)`,
pass it through.
- Grep-confirm the call site of `_resolve_repo_root` before opening the PR.

**Step 2.4** — Update the load summary line to report folded rows if it is cheap
(e.g. `N rows folded via schema alias`) — optional, but observable output helps
verify the fix on the DWH.

## Test Strategy
- **Alias folding produces canonical nodes:** load a CSV containing
  `ba_tmp.foo.col1` with an alias map `{"ba_tmp": "ba"}`; assert the graph has a
  `ba.foo` table and a `ba.foo.col1` column and **no** `ba_tmp.foo` node.
- **Phantom-node elimination (the measured bug):** with the alias map, a CSV of
  only `ba_tmp.*` rows produces zero `ba_tmp.*` table nodes.
- **DDL precedence preserved:** pre-seed a ddl-sourced `ba.x` column
  (`source='ddl'`); load a CSV with `ba_tmp.x`; assert the `HAS_COLUMN` row for
  `ba.x` still reports `source='ddl'`, not `information_schema`.
- **Dedup when both present:** CSV with both `ba.x.c` and `ba_tmp.x.c` yields a
  single `ba.x.c` column (no duplicate, no `ba_tmp` node).
- **Zero-config path unchanged:** empty alias map → behaviour identical to today
  (verbatim load).

## Acceptance Criteria
- [ ] Staging-schema rows whose schema is in the alias map upsert under the
      canonical schema; no `*_tmp` phantom table nodes are created for aliased schemas.
- [ ] DDL-sourced `HAS_COLUMN` rows are never downgraded to `information_schema`
      by a folded row.
- [ ] Duplicate canonical rows collapse to one node.
- [ ] A catalog load with no resolvable aliases behaves exactly as today.

# Fix 3 — Repo-anchored query-time config (noise filter + aliases)

## Problem

`NoiseFilter.from_config()`
([`noise_filter.py:137-167`](../../src/sqlcg/server/noise_filter.py)) falls back
to `Path.cwd()` when no `repo_root` is passed. **All CLI read call sites pass
nothing:**

- [`analyze.py:160, 197, 241, 290`](../../src/sqlcg/cli/commands/analyze.py)
- [`find.py:27, 49`](../../src/sqlcg/cli/commands/find.py)

So running `sqlcg analyze upstream …` from **outside** the indexed repo silently
loses `.sqlcg.toml` noise filtering and aliases. **Measured today:** `_bck` clone
tables appeared in traces when running from the sqlcg project dir and disappeared
when running from the DWH repo dir.

The server-side tools already do the right thing in two places —
[`tools.py:1201, 1943`](../../src/sqlcg/server/tools.py) use
`NoiseFilter.from_config(repo_root=root)` where `root = _indexed_root(db) or
Path.cwd()`. The infrastructure exists; the **CLI read commands and four
remaining `tools.py` sites** (991, 1051, 1136, 2040) don't use it.

## Design

The `Repo.path` primary key is **already persisted at index time** by
`index_cmd` ([`index.py:352-360`](../../src/sqlcg/cli/commands/index.py)) and
read back by `_indexed_root(db)`
([`tools.py:276-300`](../../src/sqlcg/server/tools.py)). **No new persisted field
is required** — the surface change is making query-time consumers resolve the
root from the graph, cwd only as a last-resort fallback.

### D3.1 — How does a CLI read command get the root?

CLI read commands route reads through `run_read_routed` and do **not** hold a
backend handle. Add a tiny read helper that resolves the indexed root via the
same routed read path the commands already use:

```sql
SELECT path FROM "Repo" LIMIT 1
```

Wrap it as a function (e.g. `resolved_repo_root() -> Path` in a shared module —
`read_client.py` or a small `cli/_config_root.py`) that runs that routed read and
returns `Path(row["path"])` if present, else `Path.cwd()`. This mirrors
`_indexed_root` but over the routed read path so it works whether or not a server
is live. Confirm `run_read_routed` tolerates a graph with no `Repo` row (returns
`[]` → cwd fallback).

### D3.2 — Which consumers must change?

Evaluate and change every query-time consumer of `.sqlcg.toml`:

1. **NoiseFilter** — the 6 CLI sites (analyze ×4 at
   [`analyze.py:160, 197, 241, 290`](../../src/sqlcg/cli/commands/analyze.py),
   find ×2 at [`find.py:27, 49`](../../src/sqlcg/cli/commands/find.py)) and the 4
   unparam'd `tools.py` sites
   ([`tools.py:991, 1051, 1136, 2040`](../../src/sqlcg/server/tools.py)). Pass the
   resolved root.
2. **gain / coverage config — VERIFIED N/A, drop from scope.** Grep confirms
   [`coverage.py`](../../src/sqlcg/cli/coverage.py) and
   [`gain.py`](../../src/sqlcg/cli/commands/gain.py) contain **zero** calls to
   `NoiseFilter.from_config`, `get_presentation_prefixes`, or `get_schema_aliases`
   — all coverage queries are raw SQL via `run_read_routed` with no `.sqlcg.toml`
   dependency. There is nothing to fix here. The original Step 3.4 ("audit
   gain.py / coverage.py") and its acceptance criterion / test are removed; the
   corresponding acceptance criterion below is replaced with a "verified N/A"
   statement. Step 3.4 is repurposed below for the `_resolve_repo_root`
   de-duplication follow-up from Fix 2 D2.1. If a future sprint adds presentation
   prefixes to `gain`, that sprint must wire `resolved_repo_root()` itself — do
   not invent that surface here.
3. **`catalog load` (Fix 2)** — already covered by Fix 2 D2.1 (standalone helper,
   de-duplicated in this PR's Step 3.4).

### D3.3 — Fallback semantics

- Repo indexed → use `Repo.path`.
- No `Repo` row (test-only / un-indexed graph) → `Path.cwd()` (unchanged
  behaviour for that case).
- Document that this is a behavioural change: queries from outside the repo now
  pick up the repo's `.sqlcg.toml`. This is the intended fix.

### D3.4 — Multi-repo: more than one `Repo` row

`resolved_repo_root()` and `_indexed_root(db)` both use `LIMIT 1` (matching the
existing `_indexed_root` convention — see `tools.py:276-300`'s docstring "first
Repo node wins (LIMIT 1)"). One DuckDB graph is conventionally one indexed repo
(`sqlcg index <path>` writes one `Repo` row keyed by `abs_path`); a second
`index` of a different path adds a second `Repo` row with no defined ordering.
This PR does **not** change that pre-existing multi-repo ambiguity — it is
out of scope. Document in the PR body: "if a graph indexes >1 repo root, config
resolution picks an arbitrary one (`LIMIT 1`, pre-existing behaviour, unchanged
by this fix)." Do not add new logic to disambiguate.

For an **un-indexed scratch dir** (the case in the gate question): if `db init`
was run but `index` never was, `Repo` has zero rows, so `resolved_repo_root()`
returns `Path.cwd()` — i.e. config resolves from wherever the CLI is invoked,
identical to today's behaviour for that case. This is the "No-Repo graph falls
back to cwd" test below.

## Implementation Steps

**Step 3.1** — Add `resolved_repo_root()` (routed read of `Repo.path`, cwd
fallback). Files: new helper module or `read_client.py`. Grep-confirm call sites
before PR.

**Step 3.2** — Update the 6 CLI `NoiseFilter.from_config()` calls (analyze ×4,
find ×2) to `NoiseFilter.from_config(repo_root=resolved_repo_root())`.

**Step 3.3** — Update the 4 unparam'd `tools.py` `from_config()` calls
([`tools.py:991, 1051, 1136, 2040`](../../src/sqlcg/server/tools.py)) to use
`_indexed_root(db) or Path.cwd()` (the pattern the other two sites already use —
they have a backend handle, so reuse `_indexed_root`, not the routed helper).

**Step 3.4** — De-duplicate Fix 2's local `_resolve_repo_root(backend)` helper in
`catalog.py` (added by PR 2; see Fix 2 D2.1's "Follow-up for PR 3"). Replace its
body with a call to `_indexed_root(db)` (import from `tools.py`, or extract

# Fix 4 — Metrics / UX polish (bundle 4a + 4b in one PR)

## Fix 4a — Exclude `kind='cte'` from the blindspot ranking

### Problem

After the dialect fix, the entire `gain` Section G "Top blindspot tables" top-10
is CTE-named structural nodes (`cte_insert` 698, `final` 322, `uitlever` 200, …),
burying the real gaps. The **scoped KPI already excludes** CTE/derived dst nodes
(`_Q_EDGE_HEALTH_SCOPED`,
[`coverage.py:71-83`](../../src/sqlcg/cli/coverage.py)); the blindspot report
must match.

### Design

`_Q_BLINDSPOT_WEIGHTED`
([`coverage.py:138-162`](../../src/sqlcg/cli/coverage.py)) ranks bad edges by
stripped dst table but does **not** exclude CTE/derived dst tables. Add the same
exclusion the scoped query uses — a `NOT EXISTS (SELECT 1 FROM "SqlTable" t WHERE
t.qualified = <stripped dst> AND t.kind IN ('cte','derived'))` guard in the
`bad_edges` CTE. The scoped KPI excludes both `cte` and `derived`; for parity the
blindspot ranking should exclude the **same set** (`'cte','derived'`), not just
`'cte'` — keep them aligned so the two numbers tell a consistent story. Note this
in the PR (the finding said `kind='cte'`; we widen to match the scoped KPI's
`('cte','derived')` for consistency).

### Implementation Steps
**Step 4a.1** — Add the kind-exclusion guard to the `bad_edges` CTE in
`_Q_BLINDSPOT_WEIGHTED`. Reuse the `_DST_TABLE` expression already in the module.
- Files: [`coverage.py`](../../src/sqlcg/cli/coverage.py)

### Test Strategy
- **CTE nodes excluded from ranking:** seed a graph with bad edges to a
  `kind='cte'` table and to a `kind='table'` table; assert `top_blindspot_tables`
  contains the real table and **not** the CTE node.
- **`blindspot_tables_for_80pct` recomputed on the filtered set:** assert the
  80%-coverage count reflects the post-exclusion volume.

## Fix 4b — Fire the hint on post-filter-empty results

### Problem

In `analyze upstream`/`downstream`
([`analyze.py:145-162, 182-199`](../../src/sqlcg/cli/commands/analyze.py)), the
bare-name fallback hint fires only when the canonical query returns **zero edges
pre-noise-filter**. When the canonical query returns edges but the noise filter
removes **all** of them, the user gets a silent empty table with no explanation.

### Design

After noise filtering, if `results` is non-empty pre-filter but empty
post-filter, print an explanatory notice — e.g.
`"All N edge(s) were removed by the noise filter (run with --raw to see them)."`
Additionally, when the canonical query itself was empty *and* the bare-name
fallback also yields nothing post-filter, keep current behaviour (empty table).
Keep the existing pre-filter bare-name hint intact; this adds a **second**
notice for the post-filter-empty case.

Both `upstream` and `downstream` carry the same block — factor the
post-filter-empty notice into a tiny shared helper to avoid drift (both functions
already duplicate the bare-name hint block; the helper keeps the two in sync).

### Implementation Steps
**Step 4b.1** — Capture `pre_filter_count = len(results)` before
`_filter_column_results`; after filtering, if `pre_filter_count > 0` and
`len(results) == 0`, print the "all N noise-filtered" notice.
- Files: [`analyze.py`](../../src/sqlcg/cli/commands/analyze.py)
**Step 4b.2** — Apply identically to `downstream` (shared helper).

### Test Strategy
- **All-noise-filtered notice:** a ref whose only edges are to `*_bck` tables —
  assert the notice text appears and `--raw` still shows the rows.
- **Normal non-empty trace:** the notice does **not** fire when surviving rows
  exist.
- **Genuinely-empty trace:** unchanged "No results" path, no spurious notice.

## Acceptance Criteria (Fix 4)
- [ ] `gain` Section G blindspot ranking excludes `cte`/`derived` dst tables,
      matching the scoped KPI.
- [ ] `analyze upstream`/`downstream` print a notice when every result was
      noise-filtered, instead of a silent empty table.
