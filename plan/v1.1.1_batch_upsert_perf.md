# Feature Plan: v1.1.1 — Batch-Upsert Perf + Relative-Path Bugfix + Upsert Guard

## Summary

A v1.1.0-line perf + bugfix batch on `feat/cluster-b-provenance`, from a live `../dwh`
verification (1,340 files: 224.6s total, **upsert = 169.3s / 75%**). The upsert pass
fires ~13,400 backend round-trips (10 `execute()` per file) when it should fire ~270
(10 per *batch*). Plus three independent small fixes: a relative-path crash in the
`index` git-walk branch, a perf-guard test pinning the new per-batch invariant, and two
low-priority code-review cleanups.

The campaign respects every entry in the CLAUDE.md **Performance invariants** table.
In particular the `_upsert_parsed_file`-uses-bulk invariant is *strengthened*, not
weakened: the new batch-level upsert still uses `upsert_nodes_bulk` / `upsert_edges_bulk`
exclusively, never `upsert_node` / `upsert_edge`.

---

## Scope

### In Scope
- **F1 (PRIMARY, perf)**: refactor the `index_repo` upsert pass so row dicts accumulate
  across all files in a batch, dedup once per batch, and flush with each `upsert_*_bulk`
  called **once per batch** (~10 `execute()` per batch → ~270 total instead of ~13,400).
  Same MERGE semantics, same per-batch transaction model, same per-file failure isolation.
- **F1 also covers** `resync_changed` (`indexer.py:730-765`), which today inlines the same
  per-file flush loop — it must share the new batch method so the win and the invariant
  apply there too.
- **F2 (bug)**: `sqlcg index <relative-path>` must not crash in the ignore-matcher.
- **F3 (test, user-requested)**: a deterministic guard pinning "upsert `execute()` count
  scales with batch count, not file count" + a new CLAUDE.md invariant-table row.
- **F4 (minor cleanup)**: W1 external-consumer warning/skip inconsistency; W2
  `_assert_indexed` no-Repo fallback comment/debug-log.

### Non-Goals
- **`reindex_file` / `sqlcg watch` path** (`indexer.py:775-799`): a separate short
  per-file transaction, one file per call by design. NOT refactored to batch — there is
  no batch to accumulate. Explicitly out of scope. (It still benefits indirectly: it
  calls `_upsert_parsed_file`, which is preserved unchanged as a single-file builder; see
  F1 design.)
- **CREATE fast-path (skip MERGE existence-checks on a fresh DB)**: deferred to a
  separate follow-up PR (PR-5, OPTIONAL, riskier). Planned below but explicitly NOT
  bundled into F1. Do not start it without F1 merged and measured.
- **Snowflake ACCESS_HISTORY query-history fusion** (v2 backlog) — untouched.
- **Changing graph output / edge set for any repo size.** The batch refactor must
  produce byte-identical graph contents to the current per-file path (CLAUDE.md design
  rule: never degrade the small-repo experience). A 20-file repo gets the same nodes,
  edges, and properties it gets today.

## Design

### F1 — Batch-scoped row accumulation + single flush per batch

#### Root cause (verified locations)
- [`_flush_batch`](src/sqlcg/indexer/indexer.py) (`indexer.py:290`) opens one transaction
  per batch of `batch_size` (default 50) files — the documented PERF-BATCH commit win.
- Inside the txn it calls [`_upsert_parsed_file`](src/sqlcg/indexer/indexer.py)
  **once per file** (`indexer.py:304-310`), and each call fires **10 `execute()`**
  round-trips (`indexer.py:1115-1135`: 4 `upsert_nodes_bulk` + 6 `upsert_edges_bulk`).
- Net: `1340 files × 10 ≈ 13,400` `execute()` calls but only ~27 COMMITs. The "bulk"
  methods are bulk *within one file*, not *within the batch*.
- `KuzuBackend.upsert_nodes_bulk` / `upsert_edges_bulk`
  ([`kuzu_backend.py:210`](src/sqlcg/core/kuzu_backend.py),
  [`:245`](src/sqlcg/core/kuzu_backend.py)) re-submit the query string on every call
  via `self._conn.execute(query, params)` — no `prepare()`/plan reuse — so each of the
  13,400 calls re-plans Cypher. For small DDL files, planning + dispatch dwarfs the write.
- Phase-B dedup (`indexer.py:1110-1112`) only dedups rows *within a single file*; a
  shared dimension table referenced by hundreds of files emits a TABLE row + MERGE in
  every one of them (cross-file duplication, redundant MERGE work).

#### The split: builder (per file) vs flusher (per batch)

`_upsert_parsed_file` currently does three things: (A) build rows from a `ParsedFile`,
(B) dedup within the file, (C) flush via 10 `execute()`. The refactor splits the seam
between row-building and flushing:

1. **New method — `_build_file_rows(parsed, defined_table_registry) -> FileRowSet`**
   - Does exactly today's Phase A (`indexer.py:890-1105`): builds the four node-row lists
     (`file_rows`, `table_rows`, `column_rows`, `query_rows`) and the six edge-row lists
     (`defined_in_edges`, `has_column_edges`, `query_defined_in_edges`,
     `selects_from_edges`, `column_lineage_edges`, `star_source_edges`), plus the
     `counts` dict for one file.
   - **No `db` access. No `execute()`.** Pure transform `ParsedFile -> rows`. This is what
     makes per-file failure isolation cheap: a parse/build error throws here, before any
     row joins the shared batch buffer.
   - `FileRowSet` is a small `@dataclass` (10 row-lists + `counts` + `parse_quality_key`).
     Define it in `indexer.py` next to the indexer.

2. **New method — `_flush_row_batch(db, batch_rows: BatchRowBuffer) -> None`**
   - Receives a `BatchRowBuffer` (the 10 accumulated cross-file row-lists), runs the
     **batch-scoped** dedup (extends today's Phase B from file-scope to batch-scope; same
     key per group — `qualified` for tables, `id` for columns/queries, `path` for files,
     `(src_key, dst_key)` (+ rel-discriminating props where present) for edges), then
     fires the **10 `upsert_*_bulk` calls exactly once** in the same dependency order as
     `indexer.py:1115-1135`.
   - This is the method that still satisfies the bulk-not-per-row invariant.

3. **Rewrite `_upsert_parsed_file` as a thin single-file wrapper** so `reindex_file`
   (out-of-scope path) keeps working unchanged:
   ```
   def _upsert_parsed_file(self, parsed, db, defined_table_registry=None) -> dict:
       rows = self._build_file_rows(parsed, defined_table_registry)
       buf = BatchRowBuffer.from_single(rows)
       self._flush_row_batch(db, buf)
       return rows.counts
   ```
   - **Grep-confirmed existing call site preserved**: `reindex_file` (`indexer.py:796`).
     This keeps the watch/reindex path byte-identical (one file → one flush → 10
     `execute()`), which is correct for a single-file short txn — no regression.

#### Rewrite `_flush_batch` (`indexer.py:290`) to accumulate then flush once

```
def _flush_batch(batch: list[ParsedFile]) -> None:
    if not batch:
        return
    buf = BatchRowBuffer()
    for parsed_in_batch in batch:
        try:
            rows = self._build_file_rows(parsed_in_batch, defined_table_registry)
        except Exception as exc:
            logger.warning("Failed to build rows for %s: %s — skipping", parsed_in_batch.path, exc)
            nonlocal_counts["quality"]["failed"] += 1
            continue
        buf.extend(rows)                       # accumulate good file's rows
        nonlocal_counts["tables"] += rows.counts["tables"]
        nonlocal_counts["edges"] += rows.counts["edges"]
        nonlocal_counts["star_sources"] += rows.counts.get("star_sources", 0)
        nonlocal_counts["columns_defined"] += rows.counts.get("columns_defined", 0)
        nonlocal_counts["quality"][rows.parse_quality_key] += 1
    with db.transaction():
        self._flush_row_batch(db, buf)         # ONE set of 10 execute() per batch
```

**Per-file failure isolation — addressed explicitly** (the finding flags this as the hard
part). Today, isolation is "one bad file in the txn loop is caught, batch continues."
After the merge, isolation moves *earlier*: `_build_file_rows` is pure and runs per file
inside its own try/except, so a file that throws during row-building is recorded as
`failed` and simply never contributes rows to `buf`. The accumulated good rows then flush
in one transaction. A failure *inside* `_flush_row_batch` (e.g. a backend error on the
UNWIND MERGE) now fails the whole batch rather than one file — but that is already true
today within `db.transaction()` for any error the bulk call raises, and it is the correct
trade for the 50× round-trip reduction. Note this behaviour change in the PR body:
*build-time* per-file errors stay isolated (the common case — bad SQL, parse artefacts);
*flush-time* backend errors fail the batch (rare; a backend error on a 50-file UNWIND was
already non-recoverable per-file before, because they shared one txn).

#### Batch-scoped dedup correctness (shared tables/edges)

This is the second win and a correctness requirement. Cross-file dedup with the same MERGE
key is *idempotent-equivalent* to per-file MERGE: MERGE-ing the same table N times across
N files produces exactly one node; deduping to one row before the single MERGE produces
the same one node. The keys:
- `file_rows` → `path`
- `table_rows` → `qualified` (last-writer-wins on `defined_in_file`; matches today, since
  today the later file's per-file MERGE `SET` overwrites the same property — confirm the
  dedup keeps the row whose `defined_in_file` is non-empty when one exists, to preserve
  the DEFINED_IN provenance; see Step 1.4 acceptance).
- `column_rows` → `id`
- `query_rows` → `id` (query_id is `path:index`, globally unique — no cross-file collision)
- edges → `(src_key, dst_key)` only. **Important**: `upsert_edges_bulk` generates
  `MERGE (src)-[r:rel_type]->(dst)` which matches on the (src_node, dst_node) pair,
  then `SET` overwrites properties. KuzuDB cannot hold two edges of the same type
  between the same (src, dst) pair — so two `COLUMN_LINEAGE` edges between the same
  column pair from different queries already resolve to one edge in the graph today
  (last-writer-wins on `query_id`/`transform`/`confidence`). The Python dedup key
  must be `(src_key, dst_key)` to match the graph's MERGE semantics; using a
  full-property-tuple key would send multiple rows to the UNWIND that map to the same
  MERGE match (redundant, not incorrect). Do NOT use full-property-tuple as the edge
  dedup key — it is more expensive and does not reflect the graph's actual cardinality.

**Open design point resolved**: today edges are NOT Python-deduped (only the three node
groups are, `indexer.py:1110-1112`). Within one file, duplicate edges between the same
(src, dst) pair are rare; across a batch they are common (shared `SELECTS_FROM`). Batch
edge-dedup is therefore a *new* Python-side dedup, not just a widened one. The correct key
is `(src_key, dst_key)` — which matches the `MERGE (src)-[r]->(dst)` cardinality in
`upsert_edges_bulk`. Sending duplicate (src_key, dst_key) rows in one UNWIND is
idempotent but wasteful; deduping prevents the redundant work. Acceptance criterion F1-AC4
pins this with an integration test asserting edge *count and properties* are identical
batch-vs-per-file on a fixture with a shared dimension referenced by multiple files.

#### `resync_changed` shares the new path

`resync_changed` (`indexer.py:730-765`) currently inlines two copies of the per-file flush
loop. Replace both with the same accumulate-then-`_flush_row_batch` pattern (extract a
local `_flush_resync_batch` closure mirroring `_flush_batch`, or — cleaner — hoist the
`_flush_batch` body into a private `_upsert_file_batch(self, batch, db, registry, counts)`
helper that both `index_repo` and `resync_changed` call). **Decision**: hoist to a shared
`_upsert_file_batch` method to avoid two divergent copies.
- **Grep-confirmed call sites**: `index_repo._flush_batch` body and `resync_changed`
  batch loops (`indexer.py:736`, `:752`).

#### KuzuBackend bulk methods — unchanged

`upsert_nodes_bulk` / `upsert_edges_bulk` already do `UNWIND $rows ... MERGE`. With
batch-scoped rows they now receive ~50× larger `$rows` lists and are called ~50× less
often. No signature change. (The `prepare()`/plan-reuse idea the finding mentions is
**not** in F1 scope — calling each bulk method once per batch already removes ~13,130 of
the ~13,400 re-plans. Plan reuse is a marginal extra and is folded into the optional PR-5
consideration, not F1.)

### F2 — Relative-path crash in the `index` git-walk branch

`sqlcg index ../dwh` raises `ValueError: '../dwh/.../X.sql' is not in the subpath of
'/home/.../dwh'`. Root cause:
- [`_git_sql_files(root)`](src/sqlcg/indexer/walker.py) (`walker.py:12-24`) builds paths as
  `root / f` where `root` is the *relative* input (`../dwh`), yielding `../dwh/...`.
- [`is_ignored`](src/sqlcg/utils/ignore.py) (`ignore.py:37-38`) does
  `root = Path(root).resolve()` (absolute) then `path.relative_to(root)` — the path is
  relative `../dwh/...`, the root is absolute → `ValueError`.
- **PLAN CORRECTION (reviewer-verified)**: The plan originally claimed #24's fix covered
  the rglob branch but live test confirms BOTH branches are currently broken with a
  relative root input. In the rglob case: `root.rglob()` yields paths relative to CWD
  when root is relative; `is_ignored` resolves root to absolute then `path.relative_to(root)`
  fails. The walker.py fix (resolve root once at the top of `walk_sql_files`) fixes both
  branches. Regression tests cover both (git-walk + rglob variants in test_cli_index.py).

**Fix**: resolve the root once, at the top of `walk_sql_files`, before either branch, so
both `_git_sql_files` and the `rglob` fallback operate on an absolute root and yield
absolute paths. Concretely in [`walker.py`](src/sqlcg/indexer/walker.py):
```
def walk_sql_files(root, spec, use_git=True):
    root = Path(root).resolve()   # NEW: single resolution point
    if use_git:
        git_files = _git_sql_files(root)   # cwd=root (abs); root / f → absolute
        ...
```
- `_git_sql_files` runs `git ls-files` with `cwd=root` and joins `root / f`; with `root`
  absolute the yielded paths are absolute, and `is_ignored`'s `path.relative_to(root)`
  now matches (both absolute, consistent).
- The `rglob` fallback already takes `root` from the same variable, so it inherits the
  resolution for free.
- `is_ignored`'s internal `Path(root).resolve()` (`ignore.py:37`) becomes redundant but
  is left as a defensive no-op (resolving an already-absolute path is idempotent) — do not
  remove it; other callers may pass relative roots.

**Grep-confirmed callers of `walk_sql_files`**: verify before edit with
`grep -rn "walk_sql_files" src/sqlcg` (expected: `index_repo` pass-1 walk). The resolution
is internal to the walker, so callers are unaffected behaviourally except the bug is gone.

### F3 — Upsert per-batch perf guard (user-requested)

Mirror the established pattern in
[`test_bulk_upsert_invariant.py`](tests/unit/test_bulk_upsert_invariant.py) and
[`test_perf_scaling_guard.py`](tests/unit/test_perf_scaling_guard.py): deterministic
op-counts via a spy/`MagicMock`, **never wall-clock**; N vs 2N flat-scaling assertion;
pins a NAMED regression.

**Home decision**: a dedicated `tests/unit/test_upsert_batch_invariant.py`, NOT a new axis
in `test_perf_scaling_guard.py`. Rationale: the scaling guard patches *parser* hot-ops
(`build_scope`/`qualify`/`sg_lineage`) against `AnsiParser` with no graph backend; the
upsert invariant is about *backend round-trips* and needs a `MagicMock` db. Different
fixture, different patch surface — a dedicated file keeps each guard's intent legible
(same reason `test_bulk_upsert_invariant.py` is its own file).

**The named invariant**: "the `index_repo` upsert pass calls `upsert_*_bulk` (hence
`_conn.execute`) a fixed number of times per *batch*, independent of file count."

**Test shape**:
- Build a `MagicMock` db (same `_make_mock_db` helper as the existing bulk test:
  `run_read` returns `[]`).
- Index N files in a single batch (`batch_size >= N`), record
  `mock_db.upsert_nodes_bulk.call_count + mock_db.upsert_edges_bulk.call_count`.
- Index 2N files in a single batch, record the same.
- Assert the two counts are **equal** (flat) and `<=` a fixed ceiling (e.g. 10 per batch
  — 4 node groups + 6 edge groups; allow `< 12` slack for empty groups skipped early).
  After F1, count scales with *batches*, not files: same batch → same count.
- A second assertion (regression-direction): with `batch_size=1`, N files → ~10×N calls
  (proves the spy actually tracks per-flush volume), so the flat-batch assertion is
  meaningful and not vacuous.
- Keep the existing `not upsert_node.called` / `not upsert_edge.called` assertion so the
  bulk-not-per-row invariant is co-pinned.

**CLAUDE.md invariant-table row** (add under the existing `_upsert_parsed_file` row):

| `_flush_row_batch` accumulates rows across all files in a batch and calls each `upsert_*_bulk` **once per batch** (not once per file) | `indexer.py` `_flush_batch`/`_upsert_file_batch` → `_flush_row_batch` | Measured: per-file flush = ~13,400 `execute()` on 1,340-file DWH (10/file); per-batch flush = ~270 (10/batch). v1.1.1. Guard: [`test_upsert_batch_invariant.py`](tests/unit/test_upsert_batch_invariant.py) |

Also add the file to the "Tests covering these invariants live in …" sentence.

### F4 — Minor cleanups

**W1** — [`_ingest_external_consumers`](src/sqlcg/indexer/indexer.py) (`indexer.py:1211-1219`):
an external consumer referencing a *non-presentation* table logs an advisory warning but
still appends the edge (`edge_rows.append(...)` at `:1219` runs after the warning's
`if` block without `continue`), whereas the *unknown-table* case (`:1205-1210`) does
`continue` (hard-skip). **Decision**: keep creating the edge (a non-presentation
reference is a quality smell, not invalid — the table exists), but make the message say so.
Change the warning text at `indexer.py:1215-1218` to: `"... references non-presentation
table '{target}' — edge created anyway (advisory)"`. This makes intent explicit and
matches observed behaviour; no control-flow change.
- **Why not hard-skip**: the table exists in the graph (it passed the
  `existing_tables` check at `:1205`), so the consumer relationship is real. Skipping would
  silently drop a true edge. Documented trade.

**W2** — [`_assert_indexed`](src/sqlcg/server/tools.py) (`tools.py:173-193`): the no-Repo
File-node fallback (`:189-192`) silently accepts a graph with Files but no Repo. Add a
`logger.debug(...)` on that branch noting "File nodes present but no Repo node — accepting
as test-only/partial graph; a production graph should always have a Repo node" so a
corrupted production graph is observable in debug logs, not silently accepted. Docstring
already explains the intent (`:176-178`); the debug-log makes it runtime-visible.
- Confirm `logger` exists in `tools.py` before adding (grep `^logger` / `getLogger`).

### PR-5 (OPTIONAL, follow-up, NOT in this batch) — CREATE fast-path for fresh DB

Riskier; separate PR; do **not** bundle. On a fresh/empty DB the ~49k `COLUMN_LINEAGE`
MERGE existence-probes always miss, so a `CREATE` (skipping the existence check) would be
faster. Requires (a) reliable fresh-DB detection (e.g. `MATCH (n) RETURN count(n)` == 0 at
index start, or a dedicated flag) and (b) **guaranteed cross-batch node dedup** so two
batches creating the same shared table do not produce duplicate nodes/edges (CREATE is not
idempotent). This interacts with F1's batch dedup (which is within-batch only). Plan it
only after F1 is merged and the ../dwh re-measurement shows MERGE probes are the next
bottleneck. **Blocking pre-req**: a cross-batch node registry. Listed here so it is not
forgotten; not a deliverable of v1.1.1.

- **E2E**
  - Relative-path `sqlcg index` (F2-AC), git-walk + rglob branches.

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Batch edge-dedup appears to drop a `COLUMN_LINEAGE` edge from a second query between the same column pair | Not a risk: `upsert_edges_bulk` uses `MERGE (src)-[r]->(dst)` — KuzuDB already collapses same-pair edges today (last-writer-wins on `query_id`/`transform`). Python dedup key `(src_key, dst_key)` matches this semantic. F1-AC4 equivalence test pins identical edge count batch-vs-per-file. |
| `defined_in_file` provenance lost when a shared table is deduped across files | Dedup prefers the row with non-empty `defined_in_file`; equivalence test asserts `SqlTable.defined_in_file` matches the per-file path |
| Flush-time backend error now fails a whole batch instead of one file | Documented behaviour change; build-time (common) errors stay per-file isolated; flush-time errors were already non-recoverable per-file within the shared txn |
| Larger per-batch memory (50 files' rows held before flush) | Batch already holds 50 `ParsedFile` objects in memory today; row dicts are smaller than the parsed ASTs they derive from; `batch_size` remains user-tunable; OOM history honoured (batched, not all-at-once) |
| F2 fix changes path form (relative→absolute) and breaks a downstream consumer of yielded paths | Paths are already consumed as filesystem paths internally; e2e test asserts end-to-end index works; `is_ignored`'s defensive `resolve()` left intact |
| Small-repo regression | F1-AC3 byte-identical-graph test on a 20-file repo |
